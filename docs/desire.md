# CPU Pipeline Playground

## Purpose

RISC-V風の小さな命令セットを使って、CPUのパイプラインや命令実行の様子を視覚的に表現し、短いアセンブリをステップ実行しながらCPU内部の状態変化を観察できる playground を作りたい。

教育資料として説明を読むものではなく、自分でコードを書き、少し変え、実行し、パイプライン内の動きが変わることを気持ちよく触れるツールにしたい。完全な実在CPU再現ではなく、単純に設計されたCPU仕様に対して、画面やCLIに見えている状態が正しいことを重視する。

## Context

命令セットはRISC-Vをベースに考える。これは実在ISAへの完全準拠を目指すためではなく、独自命令セットの議論を減らし、CPU実行の可視化に集中するためである。

中心に置く実行モデルは、まず in-order pipeline とする。依存による stall、forwarding、branch flush、命令ごとの latency、issue width などが見えることを重視する。out-of-order 実行、rename、ROB、reservation station などの複雑な機構は、最初の主目的には含めない。

このツールの哲学は、複雑な実CPUを雑に簡略化して雰囲気だけ見せることではない。見やすく理解しやすくなるようにCPU仕様そのものを単純に設計し、その仕様の中で命令意味論、パイプライン各段の状態遷移、hazard/stall/flush の発生理由、レジスタやメモリ更新のタイミングが正しいことを大切にする。

GUI は重要だが、core をGUI前提で設計したくない。CPU、assembler、simulator はUI非依存の core として作り、CLI でもステップ実行できるようにする。CLI はおまけではなく、core の単純さと正しさを保つための第一級の利用口である。GUI は同じ core の状態遷移や実行ログを視覚化する playground として扱う。

最初から主目的にしないものとして、OS、例外、割り込み、キャッシュ、実在ABI、完全なRISC-V互換、実在CPUのマイクロアーキテクチャ再現、out-of-order 実行がある。ただし、将来拡張できる余地は不必要に潰さない。

## Direction

後続の設計や実装では、まず「単純に定義されたCPU仕様に対して正しい core」を中心に据える。UIや演出は、その core の状態を心地よく観察し、操作できるようにするための層として扱う。

ユーザー体験としては、短いアセンブリを書き、実行コントロールでステップを進め、タイムライン上で命令が IF/ID/EX/MEM/WB を進む様子、レジスタやメモリの変化、stall/flush の理由が即座に見えることを目指す。CPU設定として issue width や latency などを触り、同じプログラムの動きがどう変わるかを観察できるとよい。

実装方針を考えるときは、GUIの見た目から core の状態構造を決めない。CLIで同じプログラムを実行・ステップ・検証できることを、設計の健全性を保つ制約として使う。
