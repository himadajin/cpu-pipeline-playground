# QEMU Reference Testing

## Purpose

RISC-V toolchain と QEMU を命令意味論の参照実装として使い、CPU Pipeline Playground の assembler / simulator が命令を正しく扱えているという信頼性を高めたい。

目的は数学的な形式証明ではない。手書きの入力値と期待値のペアに頼る代わりに、同じ入力プログラムを信頼できる外部実装で実行し、そこから得られる architectural state を期待値として使えるようにすることにある。これにより、命令追加や simulator 変更のたびに、開発者が自然なアセンブリ命令列を書くだけで回帰検出できる状態を目指す。

## Context

現在のプロダクトは、最初から完全な RISC-V 互換を目指すものではなく、RISC-V 風の小さな命令セットと単純な 5-stage in-order pipeline を観察する playground として作られている。一方で、将来的には対応命令を増やし、「RISC-V の pipeline viewer」に近づけたい。そのため、今後の命令拡張を安全に進められる検証基盤が必要になっている。

QEMU を参照にしたい対象は、このアプリ固有の pipeline 表示、stall、forwarding、flush、timeline、inspector の正しさではない。QEMU は命令実行後の architectural state、つまりレジスタ、メモリ、分岐結果など、命令意味論のオラクルとして使う。このアプリ固有の pipeline 挙動や可視化は、引き続き core の不変条件テストや UI テストで検証する。

テスト基盤は、成果物の最終品質ゲートというより、開発とテストをスムーズにするハーネスとしての品質保証である。GitHub CI 上で必ず毎回動かす重いゲートにする必要はなく、まずはローカルで継続的に実行し、通ったものを push する運用でもよい。必要なら専用 script や Docker 環境として、通常の Vitest とは分けて扱える形が望ましい。

現行命令サブセットにこだわりすぎて、不自然な変換や複雑なテストハーネスを作りたくない。自然なアセンブリ命令列を書いたら、それ自体がテストになるような設計を重視する。テスト基盤を単純で筋の良いものに保つために、必要であれば命令追加も検討してよい。

## Direction

後続の設計では、QEMU / RISC-V toolchain を参照実装にした differential testing として考える。CPU Pipeline Playground 側と参照実装側に同じ意図のアセンブリプログラムを与え、最終的な architectural state を正規化して比較できるようにする。

最初の主対象は、現行命令サブセットを中心に、命令意味論の一致を確認できるローカル実行可能なテスト基盤である。ただし、命令を追加しないことを目的化しない。テストプログラムを自然に書けるようにするため、また将来の RISC-V 互換化に向かうために、必要な命令や assembler 対応を増やすことは許容する。

この要望では、pipeline viewer 固有の stage 遷移やイベント表示まで QEMU で証明しようとしない。QEMU 参照テストは命令意味論の土台を強くする役割に集中し、その土台の上で pipeline の stall、forwarding、flush、timeline 表示などをアプリ自身のテストで検証する。

ハーネスは、開発者が読みやすいアセンブリテストを追加しやすい形にする。期待値ファイルの手作業メンテナンスや、現行 simulator に合わせた過度な特殊処理を増やすのではなく、参照実行、状態抽出、状態比較の流れが明確で保守しやすいことを優先する。
